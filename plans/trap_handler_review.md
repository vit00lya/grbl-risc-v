# Анализ `trap_handler` — найденные проблемы и варианты решений

## Общее описание

`trap_handler()` (строка 87, `src/main.cpp`) — это обработчик прерываний (`extern "C"`), который вызывается при любом прерывании на процессоре MIK32V2 (RISC-V). Он обрабатывает:
- Прерывания от GPIO (концевые выключатели) — вызов `pin_limit_vect()`
- Прерывания от UART (приём данных) — чтение байта в `line[buf_pointer]` и его фильтрация

---

## Проблема 1: Гонка данных (data race) — `line[]`, `buf_pointer`, `line_flags`

### Описание

`trap_handler` работает в контексте прерывания и **напрямую** пишет в глобальные переменные:
- `line[LINE_BUFFER_SIZE]` — буфер строки
- `buf_pointer` — указатель текущей позиции в буфере
- `line_flags` — флаги состояния строки

В то же время **основной цикл** в `protocol_main_loop()` (строка 31, `lib/grbl/src/protocol.cpp`) **читает и модифицирует** те же переменные:
```cpp
if(*line_flags & LINE_FLAG_LINE_READ){  // строка 72
    line[char_counter] = 0;             // строка 76
    // ...
    char_counter = 0;                   // строка 98
}
```

**Проблема:** Нет никакой синхронизации между прерыванием и основным циклом. Возможен сценарий:
1. Прерывание начало заполнять `line[]`, установило `buf_pointer = 5`
2. Основной цикл проверяет `LINE_FLAG_LINE_READ` — флаг ещё не установлен
3. Прерывание получает `\n`, обнуляет `buf_pointer = 0`, сбрасывает `LINE_FLAG_LINE_READ`
4. Прерывание получает новый байт, пишет в `line[0]`
5. Основной цикл видит `LINE_FLAG_LINE_READ == false` и не обрабатывает строку — **потеря данных**

Или обратный сценарий:
1. Прерывание получило `\n`, установило `LINE_FLAG_LINE_READ`
2. Основной цикл видит флаг, начинает читать `line[]`
3. В это же время приходит новый байт по UART, прерывание пишет в `line[0]` — **повреждение данных**

### Варианты решения

**A. Двухбуферная схема (double buffering)**
- Использовать два буфера: один заполняется в прерывании, второй обрабатывается в основном цикле
- Переключение буферов атомарно, без гонок
- Плюс: надёжно, классическое решение
- Минус: +200 байт RAM (LINE_BUFFER_SIZE)

**B. Флаг занятости (busy flag)**
- Ввести флаг `volatile bool line_busy`, который основной цикл устанавливает перед чтением `line[]`
- Прерывание проверяет флаг и не пишет в `line[]`, пока он установлен (или использует резервный буфер)
- Плюс: минимальные изменения
- Минус: возможна потеря байтов, если прерывание не может записать данные

**C. Переход на кольцевой буфер (ring buffer) в прерывании**
- В прерывании писать сырые байты в кольцевой буфер (как это сделано в `serial_poll_rx()`)
- Основной цикл читает из кольцевого буфера и выполняет фильтрацию/разбор строки
- Плюс: стандартный подход, нет гонок, нет потери данных
- Минус: требуется рефакторинг логики разбора строки

## Проблема 2: Неатомарный доступ к `buf_pointer` и `line_flags`

### Описание

Переменные `buf_pointer` (тип `uint8_t`) и `line_flags` (тип `uint8_t`) модифицируются в прерывании и читаются/пишутся в основном цикле. На RISC-V операции с `uint8_t` могут не быть атомарными (особенно если переменная не выровнена или компилятор использует load-modify-store).

### Варианты решения

**A. Использовать `volatile` для всех разделяемых переменных**
- `buf_pointer` и `line_flags` уже не объявлены как `volatile` (в отличие от `sys_rt_exec_state` и подобных)
- Добавить `volatile` — минимальное изменение

**B. Атомарный доступ**
- Использовать `__atomic_load_n()` / `__atomic_store_n()` или эквивалентные встроенные функции GCC для RISC-V
- Плюс: гарантированная атомарность
- Минус: зависит от toolchain

**C. Отключение прерываний при доступе из основного цикла**
- В `protocol_main_loop()` перед чтением/записью `line_flags` и `buf_pointer` отключать прерывания
- Плюс: простота
- Минус: увеличивает latency обработки прерываний

## Проблема 3: `serial_read()` вызывается из прерывания, но спроектирована для основного цикла

### Описание

Функция `serial_read()` (строка 63, `lib/grbl/src/serial.cpp`) просто вызывает `UART_ReadByte(UART_0)`. В документации к ней сказано: *"Called by main program"*. Однако в `trap_handler` она вызывается из прерывания.

В оригинальном Grbl для AVR прерывание UART пишет в кольцевой буфер, а `serial_read()` читает из него в основном цикле. Здесь же логика инвертирована: прерывание напрямую читает из UART и пишет в `line[]`.

### Варианты решения

**A. Переименовать/изолировать функцию**
- Создать отдельную функцию `serial_read_isr()`, которая вызывается только из прерывания
- Явно документировать, что `serial_read()` теперь ISR-safe

**B. Вернуться к классической схеме**
- В прерывании UART писать в кольцевой буфер (как в `serial_poll_rx()`)
- В основном цикле вызывать `serial_poll_rx()` или аналогичную функцию для разбора
- `trap_handler` должен только сохранять байт в кольцевой буфер, а не разбирать строку

## Проблема 4: `pin_limit_vect()` вызывает `mc_reset()` и `system_set_exec_alarm()` из прерывания

### Описание

Функция `pin_limit_vect()` (строка 120, `lib/grbl/src/limits.cpp`) вызывается из `trap_handler`. Внутри она может вызвать:
- `mc_reset()` — функция motion control, которая останавливает двигатели, очищает буферы и т.д.
- `system_set_exec_alarm()` — устанавливает флаг тревоги

Эти функции не были спроектированы для вызова из прерывания. `mc_reset()` может быть тяжёлой и вызывать другие функции, которые могут быть не-reentrant.

### Варианты решения

**A. Установка флага из прерывания, обработка в основном цикле**
- В `pin_limit_vect()` только установить флаг `sys_rt_exec_alarm`
- В `protocol_exec_rt_system()` обработать этот флаг и вызвать `mc_reset()` уже в контексте основного цикла
- Плюс: безопасно, минимальные изменения
- Минус: небольшая задержка между срабатыванием концевика и реакцией

**B. Проверить `mc_reset()` на ISR-safe**
- Проанализировать `mc_reset()` и все, что она вызывает, на предмет reentrancy
- Если безопасна — оставить как есть, но добавить документацию

## Проблема 5: `HAL_EPIC_Clear(0xFFFFFFFF)` — потенциально опасный сброс всех прерываний

### Описание

В конце `trap_handler` (строка 201) вызывается `HAL_EPIC_Clear(0xFFFFFFFF)`, что сбрасывает **все** флаги прерываний. Это может привести к потере прерываний, которые пришли во время обработки текущего.

### Варианты решения

**A. Сбрасывать только обработанные флаги**
- Вместо `0xFFFFFFFF` сбрасывать только те флаги, которые были обработаны в данном вызове
- Например, если сработало только UART-прерывание, сбросить только его флаг

**B. Использовать маски EPIC_CHECK**
- Проверять, какие прерывания обработаны, и сбрасывать только их
- Пример:
  ```cpp
  uint32_t clear_mask = 0;
  if (EPIC_CHECK_GPIO_IRQ()) clear_mask |= HAL_EPIC_GPIO_MASK;
  if (EPIC_CHECK_UART_0()) clear_mask |= HAL_EPIC_UART_0_MASK;
  HAL_EPIC_Clear(clear_mask);
  ```

## Проблема 6: Отсутствие проверки готовности UART перед чтением

### Описание

В `trap_handler` проверяется `EPIC_CHECK_UART_0()`, затем внутри проверяется `HAL_USART_RXNE_ReadFlag(UART_0)`. Однако между этими двумя проверками флаг RXNE может измениться (например, если данные уже были прочитаны другим обработчиком). Кроме того, нет проверки на ошибки UART (framing error, overrun error и т.д.).

### Варианты решения

**A. Проверять флаги ошибок UART**
- После чтения байта проверять флаги `HAL_USART_ORE_ReadFlag()` и `HAL_USART_FE_ReadFlag()`
- При ошибке — сбрасывать флаг и игнорировать байт

**B. Использовать буферизацию с проверкой**
- Читать байт только если RXNE установлен, и сразу сбрасывать RXNE
- Не полагаться на EPIC_CHECK как на единственную гарантию

## Проблема 7: Закомментированный код и мёртвые ветки

### Описание

В `trap_handler` присутствует:
- Закомментированный блок обработки `TIMER16_1` (строки 89-96, 182-196)
- Безусловный вызов `HAL_EPIC_Clear(0xFFFFFFFF)` находится **вне** блока `if (EPIC_CHECK_UART_0())` — строка 201, но закрывающая скобка `}` на строке 202 закомментирована, что создаёт путаницу

### Варианты решения

**A. Очистить код**
- Удалить закомментированные блоки
- Исправить форматирование фигурных скобок
